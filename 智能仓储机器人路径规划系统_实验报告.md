## 1. 涉及数据结构和相关背景

本实验围绕“智能仓储机器人路径规划系统”展开。仓库被抽象为二维网格，AGV 需要从装卸区出发，避开货架和临时障碍，到达目标货架后返回装卸区。该问题本质上是图搜索问题：每个可通行格子可以看作图中的顶点，相邻格子之间的移动关系可以看作边。

本系统主要涉及以下数据结构和算法：

- 二维数组：保存 `20 × 15` 仓库地图，每个格子使用整数表示不同元素。
- 结构体与类：封装坐标、地图、路径结果、搜索器和多车任务。
- 优先队列：A*、Dijkstra 和 CBS 中都需要按代价取出当前最优节点。
- 哈希表 / 集合：用于多车时间状态去重、约束查找和冲突检测。
- 前驱数组：保存最优路径上的父节点，用于搜索完成后回溯路径。
- A* 算法：使用曼哈顿距离作为启发函数，加快单车最优路径搜索。
- Dijkstra 算法：作为对照算法，验证 A* 的最优性。
- CBS 算法：作为增强功能，解决多 AGV 的顶点冲突和边交换冲突。

项目使用 C++17 实现核心算法，使用 JSON 作为输入输出协议；前端使用 React 实现 2D / 3D 可视化，后端使用 FastAPI 调用 C++ 可执行文件。

## 2. 实验内容

### 2.1 智能仓储机器人路径规划系统

#### 2.1.1 问题描述

某电商仓库中有自动导引车 AGV 负责搬运货物。仓库布局为网格状，包含通道、货架、临时障碍、装卸区和目标货架。机器人需要从装卸区出发，避开不可通行区域，到达目标货架取货，然后返回装卸区。

地图元素定义如下：

```text
0：可通行区域
1：货架，不可通行
2：临时障碍物，不可通行
3：装卸区域，起点 / 终点
4：目标货架，作为可抵达的取货点
```

#### 2.1.2 基本要求

本实验完成了以下基本要求：

- 创建默认 `20 × 15` 网格地图，并支持自定义地图。
- 实现标准 A* 路径规划算法。
- 启发函数使用曼哈顿距离。
- 支持 8 方向移动：上下左右和四个对角方向。
- 输出找到的最优路径、路径长度和搜索节点数量。
- 使用字符界面和 Web 图形界面展示地图、搜索过程和规划路径。

在基本要求之外，系统还实现了：

- Dijkstra 对照模式，用于验证 A* 结果。
- 往返路径规划，即“装卸区 -> 目标货架 -> 装卸区”。
- CBS 多车协同规划，支持多 AGV 避碰。
- 2D / 3D 可视化、曼哈顿热力层、搜索动画和路径动画。

#### 2.1.3 数据结构设计

##### 1. 坐标结构

`Point` 表示网格中的一个位置。为了能放入有序集合和哈希容器，重载了比较和哈希逻辑。

```cpp
struct Point {
    int row = 0;  // 行坐标
    int col = 0;  // 列坐标

    bool operator==(const Point& other) const;
    bool operator!=(const Point& other) const;
    bool operator<(const Point& other) const;
};

struct PointHash {
    std::size_t operator()(const Point& point) const;
};
```

##### 2. 地图结构

`GridMap` 保存仓库地图，并负责边界判断、通行判断和斜向移动规则。

```cpp
enum class CellType {
    Empty = 0,        // 通道
    Shelf = 1,        // 货架
    Obstacle = 2,     // 临时障碍
    LoadingArea = 3,  // 装卸区
    TargetShelf = 4   // 目标货架
};

class GridMap {
public:
    explicit GridMap(std::vector<std::vector<int>> cells);

    int rows() const;
    int cols() const;
    bool isInside(const Point& point) const;
    bool isPassable(const Point& point) const;
    bool canMove(const Point& current, const Point& next,
                 bool allow_diagonal,
                 bool prevent_corner_cutting) const;

private:
    std::vector<std::vector<CellType>> cells_;  // 地图矩阵
    std::optional<Point> start_;                // 单车模式起点
    std::optional<Point> target_;               // 单车模式目标点
};
```

其中 `canMove` 是地图规则的核心。斜向移动时，如果开启“禁止穿越墙角”，则需要同时检查两个正交相邻格子是否可通行，避免机器人从两个障碍物夹角中穿过去。

##### 3. 单车路径结果

`PathResult` 保存一次搜索的结果，既用于命令行输出，也用于前端动画。

```cpp
struct SearchTraceEntry {
    Point point;   // 当前扩展节点
    int g_cost;    // 起点到当前节点的真实代价
    int h_cost;    // 当前节点到目标点的曼哈顿估计
    int f_cost;    // f = g + h
};

struct PathResult {
    bool found = false;
    std::string message;
    std::vector<Point> path;             // 最终路径
    int path_cost = 0;                   // 路径长度
    int expanded_count = 0;              // 搜索节点数量
    std::vector<Point> expanded_order;   // 扩展顺序
    std::vector<SearchTraceEntry> search_trace;
};
```

##### 4. 多车任务结构

多车模式中，每辆 AGV 都有独立任务。CBS 输出每辆车的时间轴路径。

```cpp
struct RobotTask {
    std::string id;     // AGV 编号
    Point start;        // 起点
    Point target;       // 目标点
    bool round_trip;    // 是否返程
};

struct RobotPath {
    std::string id;
    std::vector<Point> timeline;                 // 每个时间步的位置
    int path_cost = 0;
    std::optional<int> return_start_time_step;   // 返程开始时间
};
```

#### 2.1.4 功能说明（函数、类）

##### 1. 单车搜索器

`SearchPlanner` 负责 A*、Dijkstra 和往返规划。A* 和 Dijkstra 共用同一套搜索框架，区别只在于启发函数：

```cpp
class SearchPlanner {
public:
    /**
     * @brief 执行单程路径规划
     * @param start      起点
     * @param target     目标点
     * @param algorithm  AStar 或 Dijkstra
     * @param options    是否允许斜向移动、是否禁止穿墙角
     * @return           路径、代价、搜索节点数量和扩展轨迹
     */
    PathResult plan(const Point& start,
                    const Point& target,
                    SearchAlgorithm algorithm,
                    const SearchOptions& options) const;

    /**
     * @brief 执行往返路径规划
     * @return 去程、返程和拼接后的完整路径
     */
    RoundTripResult planRoundTrip(const Point& start,
                                  const Point& target,
                                  SearchAlgorithm algorithm,
                                  const SearchOptions& options) const;
};
```

A* 的关键逻辑可以概括为：

```cpp
open_set.push(start);
while (open_set 非空) {
    current = open_set 中 f 最小的节点;
    if (current 是目标) {
        根据 parent 回溯路径;
        返回结果;
    }

    for (next : current 的 8 个相邻格子) {
        if (地图不允许移动到 next) continue;

        next_cost = current.g + 移动代价;
        if (next_cost 更优) {
            更新 gCost;
            记录 parent[next] = current;
            将 next 放入 open_set;
        }
    }
}
```

其中：

- 正交移动代价为 `1`。
- 斜向移动代价为 `2`。
- A* 中 `h = 曼哈顿距离`。
- Dijkstra 中 `h = 0`。

##### 2. 多车 CBS 规划器

`ConflictBasedSearch` 用于多 AGV 协同规划。CBS 不替代单车 A*，而是在高层处理冲突，低层仍然调用带时间步的 A*。

```cpp
class ConflictBasedSearch {
public:
    /**
     * @brief 多车协同路径规划
     * @param tasks                 多辆 AGV 的起点、目标和返程设置
     * @param search_options        单车移动规则
     * @param multi_robot_options   多车搜索上限
     * @return                      每辆车的时间轴路径和冲突处理结果
     */
    MultiRobotResult plan(const std::vector<RobotTask>& tasks,
                          const SearchOptions& search_options,
                          const MultiRobotOptions& multi_robot_options) const;
};
```

CBS 的基本过程：

```text
1. 先为每辆 AGV 独立规划路径。
2. 检查所有路径是否存在冲突。
3. 如果没有冲突，返回多车路径。
4. 如果存在冲突，生成两个约束分支：
   - 分支一限制第一辆车
   - 分支二限制第二辆车
5. 对受约束的车辆重新规划。
6. 按总代价和完工时间选择下一个约束树节点继续搜索。
```

系统检测两类冲突：

- 顶点冲突：同一时间步两辆车占用同一格。
- 边冲突：两辆车在相邻时间步迎面交换位置。

#### 2.1.5 调试分析（遇到的问题和解决方法）

##### 1. VS Code 标红但 CMake 能编译

早期 CMake 和测试都能通过，但 VS Code 中 `#include "SearchPlanner.hpp"` 等头文件标红。原因是 IntelliSense 没有读取 `cpp_core/include`。解决方法是开启 `compile_commands.json`，并配置 `.vscode/c_cpp_properties.json`，让编辑器使用真实编译参数。

##### 2. A* 路径方向不清晰

最初前端只用色块显示路径，用户无法判断路线从哪里开始、往哪里走。后来改成 SVG 八方向箭头，并将去程和返程分成不同颜色。对于重叠路径，箭头会错位显示，避免糊成一团。

##### 3. 曼哈顿热力层重复叠色

热力层最初使用半透明叠加，反复操作后容易出现数字盖数字、颜色深浅无规律的问题。后续改为每个格子只渲染一个固定标签，并且热力层只覆盖普通通道格，不覆盖货架、障碍、装卸区和目标点。

##### 4. 往返搜索重合导致返程看不见

静态地图中返程路径经常与去程重合，因此只看最终路径很难区分搜索过程。解决方法是把搜索扩展分成去程和返程两层：去程使用绿色实线框，返程使用黑色虚线斜纹框，并给当前扩展节点增加脉冲效果。

##### 5. CBS 多车交互不友好

多车任务点最初直接写入地图，容易和 `0-4` 地图语义混在一起。后来将多车起点和终点改为独立覆盖层，显示为 `S1 / T1`、`S2 / T2`。同时增加“设置任务点 / 编辑底图”模式切换，避免用户既想设置 AGV 点又想画障碍时发生冲突。

##### 6. 无法规划时显示 `[object Object]`

前端错误处理一开始只适合字符串错误。如果后端返回 Pydantic 校验对象，页面会显示 `[object Object]`。解决方法是统一解析错误对象，将 `loc` 和 `msg` 转换为可读文本。

##### 7. 3D 视图崩溃

3D 视图最初使用 React Three Fiber，线上构建后曾出现 `Cannot read properties of undefined` 的渲染异常。为降低生产环境渲染树风险，后来改为原生 Three.js 命令式渲染，React 只负责挂载容器和传入状态。这样保留了 3D 功能，同时减少了崩溃点。

##### 8. Windows 复现问题

在 Windows 复现时遇到多个环境问题：

- Python 3.14 会导致 `pydantic-core` 尝试源码编译并失败，因此 README 建议使用 Python 3.12。
- 没有安装 Node.js 时，系统没有 `npm` 命令，因此提交版 README 中补充了 Node.js LTS 下载入口。
- CMake 找不到 `nlohmann_json`，因此使用 vcpkg 安装 `nlohmann-json:x64-windows`，并在 CMake 命令中加入 `CMAKE_TOOLCHAIN_FILE`。
- MSVC 默认按中文代码页读取源文件，中文注释触发 `C4819`，后续出现大量变量未声明的误导性错误。解决方法是在 CMake 中对 MSVC 添加 `/utf-8` 编译选项。
- Visual Studio 会把可执行文件放在 `cpp_core/build/Release/agv-path-planner.exe`，后端原来只查找 Unix 风格路径。后续增加了 Windows Release / Debug 路径查找。

#### 2.1.6 总结和体会

本题看起来是一个路径规划题，但真正实现时涉及了较完整的数据结构和工程组织。A* 的关键不是简单地“找路”，而是正确维护 `gCost`、优先队列和 `parent`，并保证启发函数不会高估真实代价。由于本题要求八方向移动，因此将斜向代价设为 `2`，曼哈顿距离仍然可以保持可采纳性。

多车 CBS 的难点在于加入时间维度后，状态空间明显变大。单车路径只需要考虑坐标，而多车路径还需要考虑时间步、等待动作和约束条件。因此必须设置最大车辆数、最大高层节点数和最大时间步，避免极端输入导致搜索时间失控。

前端可视化也反过来帮助检查算法。例如扩展节点顺序、曼哈顿热力层和 `g / h / f` 面板可以直观看出 A* 是否按预期工作；CBS 时间轴动画可以检查是否存在同格冲突和迎面交换冲突。

## 3. 实验总结

本实验完成了一个可运行的智能仓储 AGV 路径规划系统。C++ 核心实现了地图建模、A*、Dijkstra、往返规划和 CBS 多车协同；Web 前端实现了地图编辑、路径可视化、搜索动画和 3D 仓库展示；FastAPI 后端负责连接前端和 C++ 核心。

从数据结构角度看，本实验综合使用了数组、结构体、优先队列、哈希表、集合和图搜索思想。A* 和 Dijkstra 体现了图最短路问题的经典解法，CBS 则展示了如何在单车搜索的基础上扩展到多智能体协同。

本实验的主要收获是：算法正确性和工程可复现性同样重要。除了实现核心算法，还需要处理 JSON 协议、跨平台构建、依赖安装、前端交互和错误提示等问题。尤其是在 Windows 复现过程中，Python 版本、CMake 依赖、MSVC 编码和可执行文件路径都可能影响最终能否运行。通过逐步定位和修复这些问题，系统最终形成了较完整、可测试、可演示的课程项目。
