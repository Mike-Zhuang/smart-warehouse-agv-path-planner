# 数据结构期末作业代码实现 Plan

项目题目：**智能仓储机器人路径规划系统**

选题：第 4 题  
定位：用 **C++ 实现核心数据结构与 A\* 路径规划算法**，用 **Python / Web 前端做交互与可视化**。本阶段只规划代码实现，不展开实验报告和部署内容。

---

## 1. 总体实现目标

这个项目不要做成单纯的“控制台输入地图、输出路径”。更合适的实现形态是：

> C++ 算法核心 + Python 接口层 + Web 可视化前端

核心评分点仍然放在 C++ 数据结构和算法实现上，前端只是把算法过程和结果展示出来。这样既满足数据结构课程要求，也能让项目有更强的完成度。

整体目标包括：

1. 支持 20 × 15 网格地图，也允许调整地图尺寸。
2. 支持五类地图元素：
   - `0`：可通行区域
   - `1`：货架，不可通行
   - `2`：临时障碍物，不可通行
   - `3`：装卸区域，作为起点 / 终点
   - `4`：目标货架 / 目标取货点
3. 实现标准 A\* 路径规划算法。
4. 启发函数使用曼哈顿距离。
5. 支持 8 方向移动。
6. 输出最优路径、路径长度、搜索节点数量。
7. 支持路径可视化和搜索过程可视化。
8. 支持从装卸区出发，到达目标货架，再返回装卸区。

## 3. C++ 核心算法模块设计

### 3.1 基本数据结构

#### 3.1.1 `Point`

表示网格中的一个坐标点。

字段设计：

```text
row：行坐标
col：列坐标
```

主要功能：

1. 判断两个点是否相等。
2. 判断点是否合法。
3. 作为路径数组中的基本元素。
4. 作为父节点回溯时的坐标记录。

坐标建议统一使用：

```text
左上角为 (0, 0)
row 向下增加
col 向右增加
```

这样和二维数组 `grid[row][col]` 保持一致。

---

#### 3.1.2 `CellType`

定义地图元素类型。

```text
EMPTY = 0       可通行区域
SHELF = 1       普通货架，不可通行
OBSTACLE = 2    临时障碍物，不可通行
START = 3       装卸区域，起点 / 终点
TARGET = 4      目标货架 / 目标取货点
```

实现时可以用 `enum class CellType`，避免直接在逻辑中大量写魔法数字。

---

#### 3.1.3 `GridMap`

负责保存和管理地图。

核心字段：

```text
rows：地图行数
cols：地图列数
grid：二维数组，保存每个格子的类型
start：装卸区坐标
target：目标货架坐标
```

核心方法：

```text
loadFromMatrix(matrix)
getCell(row, col)
setCell(row, col, type)
isInside(point)
isPassable(point)
findStart()
findTarget()
validateMap()
```

`isPassable(point)` 的判断逻辑建议如下：

```text
0 可通行
3 起点可通行
4 目标点可通行
1 货架不可通行
2 障碍物不可通行
```

这里有一个需要主动设计的地方：题目写的是“目标货架”，但又写“货架不可通行”。为了让算法符合题目要求，可以把 `4` 理解为“目标货架对应的取货点”，即 AGV 可以到达这个格子。普通货架 `1` 仍然不可通行。

如果想更严谨，可以在增强版本里把 `4` 作为不可通行货架本体，然后让机器人到达它的相邻通道格。但基础版本建议先把 `4` 当作可到达目标点，避免不必要的复杂化。

---

### 3.2 A\* 节点结构设计

#### 3.2.1 `SearchNode`

A\* 中每个搜索节点需要保存：

```text
point：当前坐标
g：从起点到当前点的实际代价
h：从当前点到目标点的启发式估计
f：f = g + h
```

不建议在 `SearchNode` 里保存完整路径，因为这样会导致大量路径复制。更好的方式是：

```text
每个格子只保存自己的 parent
最后从 target 反向回溯到 start
```

---

#### 3.2.2 Open Set

Open Set 用最小堆实现。

C++ 中可以用：

```text
priority_queue<SearchNode, vector<SearchNode>, CompareNode>
```

默认 `priority_queue` 是大顶堆，所以需要自定义比较器，让 `f` 最小的节点优先弹出。

比较规则建议：

1. `f` 小的优先。
2. 如果 `f` 相同，`h` 小的优先。
3. 如果 `f` 和 `h` 都相同，可以按坐标稳定排序，保证结果可复现。

这样路径结果会更稳定，前端展示时不会每次路径形态都略有不同。

---

#### 3.2.3 Closed Set

Closed Set 用二维布尔数组表示：

```text
closed[rows][cols]
```

含义：

```text
closed[r][c] = true
表示这个格子已经被正式扩展过
```

同时需要一个二维数组记录当前已知最小代价：

```text
gCost[rows][cols]
```

初始化为无穷大，起点为 0。

还需要一个二维数组记录父节点：

```text
parent[rows][cols]
```

每个格子保存它在当前最优路径中的前驱点。

## 4. 方向移动与代价设计

题目要求支持 8 方向移动，即：

```text
上、下、左、右
左上、右上、左下、右下
```

可以定义方向数组：

```text
(-1,  0) 上
( 1,  0) 下
( 0, -1) 左
( 0,  1) 右
(-1, -1) 左上
(-1,  1) 右上
( 1, -1) 左下
( 1,  1) 右下
```

### 4.1 推荐主实现：曼哈顿启发 + 整数代价

题目明确要求“启发函数使用曼哈顿距离”，同时要求“8 方向移动”。为了让二者不冲突，建议主实现采用下面的代价设计：

```text
上下左右移动代价 = 1
对角线移动代价 = 2
启发函数 h = |x1 - x2| + |y1 - y2|
```

这样曼哈顿距离是可采纳的，也更容易保证 A\* 的最优性。

原因是：

```text
一次上下左右移动最多让曼哈顿距离减少 1，代价也是 1
一次对角线移动最多让曼哈顿距离减少 2，代价也是 2
```

因此启发函数不会高估真实代价。

### 4.2 可选增强：展示模式中的真实几何代价

如果希望前端展示更接近实际路径长度，也可以额外提供一个“几何展示长度”：

```text
上下左右长度 = 1
对角线长度 = 1.414
```

但注意，这个只作为展示统计，不作为主 A\* 的 `g` 代价。主算法仍然用题目要求的曼哈顿启发体系。

这样处理比较稳：既遵守题目，又不会在算法正确性上留下明显漏洞。

## 5. 对角线移动的障碍处理

路径规划中有一个常见问题：机器人是否允许“贴着两个障碍物的角落斜穿过去”。

例如：

```text
当前位置 A
右侧是障碍
下方是障碍
右下角是空地
```

如果允许从 A 直接斜着走到右下角，就会出现“穿墙角”的问题。

建议采用更合理的规则：

```text
从 (r, c) 斜向移动到 (r + dr, c + dc) 时，
不仅要求目标格子可通行，
还要求 (r + dr, c) 和 (r, c + dc) 两个相邻正交格子也可通行。
```

也就是说，不允许穿过两个障碍物夹出来的墙角。

这个规则可以作为 `GridMap` 或 `AStarPlanner` 中的一个函数：

```text
canMove(current, next)
```

逻辑包括：

1. `next` 在地图范围内。
2. `next` 可通行。
3. 如果是对角线移动，额外检查两个正交邻居是否可通行。

## 6. A\* 主流程设计

### 6.1 单程路径规划

单程路径指：

```text
装卸区 -> 目标货架
```

主流程：

```text
1. 检查地图是否合法
   - 是否存在唯一的起点 3
   - 是否存在唯一的目标点 4
   - 起点和目标点是否可通行

2. 初始化 open set
   - 起点 g = 0
   - 起点 h = Manhattan(start, target)
   - 起点 f = g + h
   - 将起点加入 open set

3. 初始化辅助数组
   - gCost 全部设为 INF
   - parent 全部设为空
   - closed 全部设为 false
   - gCost[start] = 0

4. 当 open set 不为空时循环：
   4.1 弹出 f 最小的节点 current
   4.2 如果 current 已经在 closed 中，跳过
   4.3 将 current 标记为 closed
   4.4 记录 current 到 expandedOrder，用于前端动画
   4.5 如果 current 是 target，搜索成功，停止循环
   4.6 遍历 current 的 8 个邻居
       - 判断邻居是否在地图内
       - 判断邻居是否可通行
       - 判断是否允许对角线移动
       - 计算 tentativeG = gCost[current] + moveCost
       - 如果 tentativeG < gCost[neighbor]
           更新 gCost[neighbor]
           更新 parent[neighbor] = current
           计算 h 和 f
           将 neighbor 加入 open set

5. 如果找到 target：
   - 从 target 根据 parent 反向回溯到 start
   - 反转路径
   - 输出路径、总代价、扩展节点数

6. 如果没有找到 target：
   - 返回 found = false
   - 输出已经搜索过的节点数量
   - 前端显示“无法到达目标”
```

---

### 6.2 返回路径规划

题目要求机器人从装卸区域出发，到达指定货架，然后返回装卸区域。

最简单的做法：

```text
第一段：start -> target
第二段：target -> start
```

虽然在静态地图中，第二段通常等于第一段路径反向，但建议仍然调用一次 A\*：

```text
outbound = plan(start, target)
returnPath = plan(target, start)
```

这样有两个好处：

1. 逻辑更完整，符合“去程 + 返程”的任务设定。
2. 后续如果增加动态障碍物，返程可以重新规划。

最终结果可以组合成：

```text
fullPath = outbound.path + returnPath.path 去掉重复 target 点
```

结果中保留：

```text
outboundPath：去程路径
returnPath：返程路径
fullPath：完整路径
outboundCost：去程代价
returnCost：返程代价
totalCost：总路径代价
expandedNodes：两次搜索扩展节点总数
```

前端可以用不同线型或动画顺序显示去程和返程。

## 7. 输出结果结构设计

C++ 算法核心最终应该输出一个结构化结果，而不是只打印几行文本。

建议定义 `PathResult`：

```text
found：是否找到路径
message：状态信息
path：最终路径坐标数组
pathCost：路径总代价
expandedCount：扩展节点数量
expandedOrder：节点扩展顺序
closedMap：最终 closed set
```

对于“去程 + 返程”任务，可以定义 `RoundTripResult`：

```text
success：整体任务是否成功
outbound：去程 PathResult
returnTrip：返程 PathResult
fullPath：完整路径
totalCost：去程 + 返程总代价
totalExpandedCount：两次搜索扩展节点数之和
```

输出给前端时建议使用 JSON，例如：

```json
{
  "success": true,
  "totalCost": 42,
  "totalExpandedCount": 138,
  "outbound": {
    "found": true,
    "pathCost": 21,
    "expandedCount": 69,
    "path": [
      [12, 2],
      [11, 3],
      [10, 4]
    ]
  },
  "returnTrip": {
    "found": true,
    "pathCost": 21,
    "expandedCount": 69,
    "path": [
      [10, 4],
      [11, 3],
      [12, 2]
    ]
  }
}
```

这不是要求你现在写代码，而是建议提前把接口格式想清楚。接口格式一旦稳定，C++、Python、前端可以并行开发。

## 8. C++ 输入输出方式

为了让 C++ 核心可以被前端间接调用，建议不要依赖人工命令行输入。更好的方式是：

```text
C++ 程序读取 JSON 文件或标准输入
C++ 程序输出 JSON 到标准输出
```

例如：

```text
输入：
{
  "rows": 20,
  "cols": 15,
  "grid": [[...], [...]],
  "start": [18, 1],
  "target": [2, 13],
  "allowDiagonal": true,
  "preventCornerCutting": true
}

输出：
{
  "success": true,
  "path": [...],
  "expandedOrder": [...],
  "totalCost": 36
}
```

这样 Python 后端可以通过 `subprocess` 调用 C++ 程序：

```text
1. Python 接收前端发来的地图 JSON
2. Python 将 JSON 传给 C++ 程序
3. C++ 程序运行 A*
4. C++ 输出 JSON
5. Python 把结果返回给前端
```

这种方式比一开始就用 `pybind11` 更稳，也更容易调试。

## 9. Python 接口层设计

Python 层不写算法，只做三件事：

```text
1. 接收前端请求
2. 调用 C++ 算法核心
3. 返回算法结果
```

可以用 FastAPI。

### 9.1 API 设计

建议设计以下接口：

```text
POST /api/plan
功能：计算装卸区到目标货架再返回装卸区的完整路径
```

请求内容：

```text
grid：二维地图数组
rows：行数
cols：列数
options：算法选项
```

返回内容：

```text
success：是否成功
outbound：去程结果
returnTrip：返程结果
fullPath：完整路径
totalCost：总路径长度
totalExpandedCount：搜索节点总数
message：说明信息
```

还可以设计：

```text
POST /api/validate-map
功能：检查地图是否合法

GET /api/sample-maps
功能：返回示例地图列表

GET /api/health
功能：检查后端是否运行正常
```

### 9.2 后端校验

后端需要在调用 C++ 前做基本校验：

1. 地图是否为空。
2. 每一行长度是否一致。
3. 地图元素是否只包含 0、1、2、3、4。
4. 是否有且只有一个起点 `3`。
5. 是否有且只有一个目标 `4`。
6. 起点和目标是否没有被障碍覆盖。
7. 行列大小是否超过设定上限。

不过真正的算法校验仍然应该放在 C++ 中再做一遍。这样即使绕过前端和后端，单独运行 C++ 核心也不会出错。

## 10. 前端功能设计

前端目标是把算法过程展示清楚，而不是单纯做装饰。

### 10.1 主界面布局

建议分成三块：

```text
左侧：工具栏
中间：网格地图
右侧：结果面板
```

#### 左侧工具栏

包含：

```text
选择绘制类型：
- 通道
- 货架

操作按钮：
- 运行 A*

- 随机生成障碍
- 加载示例地图

- 导出 JSON
```

#### 中间网格地图

每个格子根据类型显示不同样式：

```text
0 通道：浅色

4 目标货架：橙色

```

#### 右侧结果面板


---

### 10.2 地图编辑交互

前端网格应支持点击或拖拽绘制。

基本规则：

1. 选择一种格子类型。
2. 点击网格即可修改该格子。
3. 起点 `3` 全图只能有一个。
4. 目标 `4` 全图只能有一个。
5. 如果用户设置新的起点，旧起点自动变回通道。
6. 如果用户设置新的目标，旧目标自动变回通道。
7. 路径运行后，如果继续编辑地图，应清除旧路径结果。

这样交互会比较顺，不需要用户手动处理冲突。

---

### 10.3 搜索过程动画

C++ 返回 `expandedOrder` 后，前端可以逐个播放节点扩展过程。

动画逻辑：

```text
1. 用户点击“运行 A*”
2. 前端请求后端
3. 后端返回结果
4. 前端先播放 expandedOrder
5. 再高亮最终 path
6. 如果是往返路径：
   - 先播放去程搜索过程
   - 显示去程路径
   - 再播放返程搜索过程
   - 显示返程路径
```

动画速度可以设置：

```text
慢速：100 ms / node
中速：40 ms / node
快速：10 ms / node
立即显示：不播放动画
```

这部分不影响算法，但展示效果很明显。

### 11.2 复杂绕行地图

特点：


## 15. 代码实现中的重点和难点

### 15.1 重点一：A\* 的数据结构实现

老师最容易看出水平的地方是：

```text
priority_queue
gCost 二维数组
parent 二维数组
closed 二维数组
路径回溯
```

这些应该写得清楚、规范，不要把所有逻辑堆在一个函数里。

---

### 15.2 重点二：地图建模

地图不是简单二维数组，还要有语义：

```text
哪些格子可通行
哪些格子不可通行
哪里是起点
哪里是目标
对角线是否允许
是否允许穿墙角
```

建议把这些封装进 `GridMap` 和 `AStarPlanner`，不要散落在前端或后端。

---

### 15.3 重点三：结果结构化

不要只打印：

```text
Path found!
```

而是要输出：

```text
路径坐标
路径长度
扩展节点数量
搜索顺序
是否成功
失败原因
```

这样前端才能做可视化，报告也能写得有依据。

---

### 15.4 难点一：曼哈顿距离与 8 方向移动的关系

这是本题最容易被忽略的理论点。

建议主实现采用：

```text
上下左右代价 = 1
对角线代价 = 2
h = 曼哈顿距离
```

这样能保证题目要求和最优性之间不冲突。

如果实现成：

```text
上下左右代价 = 1
对角线代价 = 1 或 1.414
h = 曼哈顿距离
```

那么曼哈顿距离可能会高估真实代价，A\* 的最优性理论上不再严格成立。

---

### 15.5 难点二：前端动画与算法结果同步

动画不要让前端重新算算法。正确方式是：

```text
C++ 返回 expandedOrder
前端只负责按顺序播放
```

这样可以保证：

```text
前端展示的过程就是 C++ 算法真实搜索过程
```

---

### 15.6 难点三：无路径情况

无路径不是程序错误，而是合法结果。

C++ 应返回：

```text
found = false
message = "No feasible path found"
expandedCount = 已扩展节点数
expandedOrder = 已搜索节点序列
```

前端应显示搜索失败，但仍然可以播放搜索过程。

## 21. 附录：原始题目

★★★智能仓储机器人路径规划系统
某电商公司拥有一个大型自动化仓库，仓库内有多台自动导引车(AGV)负责货物搬运。仓库布局为网格状，包含：
货架区域：机器人需要到达指定货架取货
通道：机器人移动的主要路径
装卸区域：货物装载和卸载的位置
障碍物：临时堆放的货物、维修区域等。现在你是一名机器人算法工程师，需要为仓库中的AGV设计一个高效的路径规划系统。机器人需要从起始位置（装卸区域）出发，避开障碍物，到达指定的货架位置，然后返回装卸区域。

要求
（1）创建一个20×15（可根据自己的情况适当调整大小）的网格地图表示仓库布局，定义以下地图元素：
0：可通行区域
1：货架（不可通行）
2：障碍物（临时堆放）
3：装卸区域（起点/终点）
4：目标货架
（2）实现标准的A\*路径规划算法，启发函数使用曼哈顿距离，支持8方向移动（上下左右+对角线），记录并输出找到的最优路径
（3）路径可视化，使用图形界面或字符界面显示地图，显示规划出的路径，显示路径长度和搜索节点数量。
