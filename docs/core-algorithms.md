# 核心算法说明

## 1. 单车路径规划

本项目使用网格地图描述仓库。货架和临时障碍物不可进入，装卸区、通道和目标取货点
可以进入。机器人支持上下左右和斜向移动，但默认禁止穿过障碍物形成的墙角。

### 1.1 A*

A* 会优先扩展“已经付出的代价”和“预计剩余代价”之和较小的节点：

```text
f = g + h
```

- `g`：从起点到当前节点的实际代价。
- `h`：从当前节点到目标节点的估计代价。
- `f`：用于决定搜索顺序的综合代价。

项目使用曼哈顿距离作为 `h`。正交移动代价为 `1`，斜向移动代价为 `2`。一次斜向
移动最多减少两格曼哈顿距离，同时也需要付出 `2` 的代价，因此启发函数不会高估
真实代价。这个性质使 A* 能够保持最优性。

网页可以把曼哈顿距离直接画成热力层：每个可通行格子的数值表示它到当前目标的 `h`。
热力层只负责帮助理解算法，不会改变优先队列、路径代价或最终规划结果。右侧观察面板
使用核心程序输出的搜索轨迹展示 `g`、`h` 和 `f = g + h`，因此能够对应真实扩展顺序。

### 1.2 Dijkstra

Dijkstra 与 A* 可以复用同一套搜索框架。区别只有一个：Dijkstra 不使用剩余距离
估计，相当于始终令 `h = 0`。

两种算法应得到相同的最优路径代价。通常 A* 扩展的节点更少，因此对比
`expandedCount` 可以直观体现启发函数的价值。

### 1.3 关键数据结构

- 最小堆：快速取出当前最值得扩展的节点。
- `gCost`：保存每个格子目前已知的最小实际代价。
- `closed`：标记已经正式扩展的格子，避免重复处理。
- `parent`：记录最优路径上的前驱节点，搜索完成后用于反向回溯。
- `expandedOrder`：保存节点扩展顺序，后续网页可以据此播放搜索动画。

## 2. 往返路径

单车任务由两次搜索组成：

```text
装卸区 -> 目标取货点 -> 装卸区
```

去程和返程应分别运行搜索算法。静态地图中返程往往是去程的反向路径，但独立搜索
可以为后续加入动态障碍物保留空间。拼接完整路径时需要去除重复的目标节点。

## 3. 多车协同规划

多车模式使用 CBS（Conflict-Based Search）。它将路径规划拆成高层冲突处理和低层
单车搜索两部分。

### 3.1 时间轴

低层搜索不能只记录坐标，还要记录时间步。机器人允许原地等待，这样才能在狭窄通道
中主动让行。往返任务使用同一条连续时间轴：机器人抵达目标后切换到返程阶段，最终
回到自己的起点。

### 3.2 冲突类型

- 顶点冲突：两辆机器人在同一个时间步占用同一个格子。
- 边冲突：两辆机器人在相邻时间步迎面交换位置。

发现冲突后，高层约束树会创建两个分支：每个分支分别限制其中一辆机器人，再重新
规划受影响机器人的路径。优先队列按照总代价和完工时间选择更值得继续探索的分支。

### 3.3 资源限制

CBS 在复杂地图中的搜索空间可能快速增长，因此请求允许配置：

```json
{
  "options": {
    "maxRobots": 8,
    "maxHighLevelNodes": 10000,
    "maxTimeSteps": 300
  }
}
```

程序同时设置硬上限，避免异常请求长时间占用服务器资源。

## 4. 复杂度与展示数据

- 单车 A* 与 Dijkstra：时间复杂度约为 `O(V log V)`，空间复杂度约为 `O(V)`。
- CBS：性能取决于冲突数量，最坏情况下搜索空间会显著增长，因此需要资源限制。
- 网页展示可以直接使用 `path`、`expandedOrder`、`timeline`、`pathCost` 和
  `resolvedConflictCount`，不需要在前端重新实现算法。
